# 협업과 애자일

  - 상당히 많은 회사들이 스크럼 마스터를 찾고 있음(미국), 스크럼을 사용하는 회사가 대단히 많음
  - 국내의 많은 기업들도 애자일을 도입

## 애자일 개념

- 폭포수 모델

  - 세부적인 기획, 문서
  - 변화에 대응하기가 어려움
  - 변수를 예상하려고 함
- 애자일

  - 무계획이 아님
    - 큰 틀에서만 계획을 잡음
  - 예상치 못한 변수의 존재를 인정하고, 짧은 미래만 계획, 빠른 실행
  - 점진적 발전
    - 한 번에 만들지 않음
    - 핵심 기능만 빠르게 구현해서 시장 반응을 빠르게 반영
  - 마시멜로 챌린지
    - 일단 하고 보는 것이 더 효율적
    - 긴 계획이 오히려 낮은 성과
    - 빠르게 피드백을 받는 것의 중요성 -> 프로토타입
    - 업무 촉진의 중요성(CEO와 비서)
- 멀티태스킹
  - 업무는 한 번에 몰아서 하는 것이 효율적

- 우선순위
  - 빠른 처리 가능, 마감 촉박
  - 빠른 처리 불가, 마감 촉박
  - 빠른 처리 가능, 마감 넉넉
  - 빠른 처리 불가, 마감 넉넉

  - 스크럼 마스터와 애자일 코치는 유사
    - 스크럼 방식에서의 애자일 코치
  - 역할
    - 스크럼과 애자일의 가치를 잊지 않게 해주는 사람
      - 가치의 상기
      - 이론적인 부분에 대한 설명
    - 보다 더 효율적이고 효과적으로 일하는 법을 고민
      - 스크럼 방식에서 생기는 많은 활동/미팅들의 시행 시점, 방식 설정, 진행자
      - 여러 이벤트들의 진행 경험 토대로 더 나은 방식을 고민
      - 협업 툴 관리, 지원, 추진
    - 헌신적인 리더(servant leader)
      - 팀 외부적으로 팀원의 보호 역할
        - 팀원이 자신들의 일에 집중할 수 있게끔 도움
      - 팀 내부적인 업무
        - 내부 분쟁 조정
- 모든 것을 아는 사람이 아님
  - 코칭을 하는 것일 뿐
  - 각 분야의 전문가는 따로 있음
  - 산파술
- 팀장과 다른점
  - 전통적 팀에서 팀장이 하던 일은 PO와 SM이 나눈다 생각



- PO

  - 다른 역할과의 비교
    - 이해관계자들의 요구 정리, 팀에 전달이라는 면에서
      - 기획자와 유사
      - 기획자들이 장기적으로 PO로 발전해 나가려 하는 경향도
    - PM?
      - Product Manager
        - 표현상의 차이만 있을 뿐 유사
      - Project Manager
        - 일정관리 측면에서는 SM과 겹치긴 함
        - SM의 주 역할에 가깝긴 함
        - 전체적인 프로젝트의 방향성 결정이라면 PM의 일

  - 역할
    - 모든 이해관계자의 입장을 대변하는 사람
      - 고객, 사용자 등
        - 고객: 지불자
        - 사용자: 서비스 이용자
      - 각자가 원하는 바를 모두 들어야 함
    - 방향을 정하고 결정을 내리는 사람
      - 무엇을 만들 것이고, 무엇을 뺄 것이고 등
      - "프로덕트의 방향성에 대한" 결정
        - 모든 결정을 하는 사람은 아님
      - 고객 인터뷰, 데이터 분석 등 시장 파악
      - 우리팀이 해결해야 할 문제를 찾는 것
    - 끊임 없이 배우는 사람
      - 해야할 일도 많고
      - 새로운 상황에 접할 일도 많음
- PO와 SM?
  - 둘은 같은 사람이 하는 역할이 아님
  - SM: 팀을 보호, PO: 팀을 재촉

- 개발팀
  - 주체적으로 일함
    - 책임도 자신에게 있음
  - 교차기능적 협업하는 T자형 인재
  - 기술적 채무를 만들지 않음

## 로드맵

- 분기별로 할 일을 3 ~ 4개씩 작성

- OKR
  - 로드맵과도 다름, 스크럼과도 별개 개념
  - Objective(목표)
    - 도전적이고 동기부여를 할 수 있는 목표
    - Key Result와 달리 정성적
  - Key Result(핵심 결과)
    - Output이 아니라 Outcome으로
      - Output은 수행만 하면 결과가 나옴: e-mail 10개 보내기
      - Outcome은 결과물이 존재: 계약 10개 따내기
    - 반드시 숫자를 포함
    - 도전적으로 작성
      - 100% 달성이 아니면 실패가 아님
      - 오히려 100% 달성이면 Key Result를 잘못 작성한 것
  - 각 개인이 모두 자신의 목표를 작성
  - 분기마다 작성하고, 평가해보았는데, 
  - KPI; Key Performance Indicator의 대안으로 등장
    - 단순 숫자 달성
    - 달성 / 미달성으로만 단순하게 나눔: 무조건 100%
- 로드맵이 Output에 가깝게 작성된다면, OKR은 Outcome 중심으로 작성됨



## 프로덕트 백로그

### 프로덕트 백로그

- 우선순위에 따른 순서 조정
- 지나치게 큰 단위의 일을 작은 조각으로 쪼개기
- 프로덕트 백로그의 주인은 프로덕트 오너
  - 프로덕트 오너가 추가, 수정, 우선순위 관리 등의 권한을 가짐
  - 프로덕트 백로그의 관리가 PO의 일이기 때문에, PO는 문제: 해결해야 할 일을 찾아내야 함

- 프로덕트 백로그를 개인 업무에 적용해보기
  - 출근하면 가장 먼저 할 일 리스트 작성
  - 우선순위에 따라 순서 조정
  - 당장 해야할 일일수록 자세하고 상세하게
  - 나중에 할 일은 대충
  - 모든 할 일은 함께 일하는 사람들과 이야기하여 결정

- **유저 스토리**
  - 프로덕트 백로그의 할 일은 유저 스토리의 형태로 작성
  - 구조
    - `고객/사용자`는 `목적/목표`를 위해 `필요/욕구`를 원한다.
  - 포스트잇 등에 작성하는 것은 짧고 간결하게 작성해야 하기 때문
  - 작성하는 이유
    - 고객의 입장에서 생각할 수 있음
    - 원래 방식보다 더 좋은 방식으로 일 하게 됨
      - 구체화
    - 복잡한 기획안을 쓸 필요가 없어짐
      - 단순히 작성한 유저 스토리를 논의 대상으로 삼을 수 있음
  - 혼자서 유저 스토리를 작성해보는 것도 효과를 볼 수 있음



### 유저스토리 워크샵 & 스토리보드

- 스크럼과는 별개
- 유저스토리 워크샵
  - 분기 당 1회 정도 PO 혼자서가 아니라 팀원 & 이해관계자들이 모여서 프로덕트 백로그의 초안을 만드는 과정
  - 세부
    - 분기에 1번 정도
    - 사무실 / 회의실 / 워크샵 등
    - 모든 이해관계자가 모여
    - PO가 프로덕트 백로그를 만들 수 있게끔
    - 다같이 유저스토리를 작성하는
    - 유저스토리 워크샵을 진행
  - 고객의 말 = 유저스토리가 아니다
    - cf.) mom test
  - cf.) LEAN 방법론

- 스토리보드
  - 유저스토리 워크샵에서 나온 유저스토리 기반으로 작성
  - 사용자 여정
    - 로그인 -> 날짜, 시간 선택 -> 영화 시간표 확인 -> 좌석 선택 -> 결제
  - 사용자 여정(시간)에 따라 유저 스토리를 정리(좌우)
  - 우선순위 순서대로 정렬(상하)
  - 스토리 보드의 첫 번째 줄
    - MVP; Minimum Viable Product
    - MMF;Minimum Marketable Feature
    - 물론 특정 여정의 경우 첫 줄이 비어 있을 수 있음



### 플래닝 포커

- TODO 작업에 걸리는 시간을 측정하기 위한 방법
- 숫자 카드들
  - 업무를 마치는 데에 소모되는 시간 단위
  - ex) 물론 1이 의미하는 단위는 달라질 수 있음
    - 0: 금방
    - 1/2: 반나절
    - 1: 하루면 된다
- 프로덕트 백로그를 각자 놔두고
  - 각 팀원이 각각의 요소 마다 걸릴 것으로 예상되는 시간을 예상하여 카드를 제시
  - 모두가 같은 숫자를 제시할 때까지 진행함
  - 보통 2, 4, 6, 7은 없음
    - 카드는 대부분의 경우 피보나치 수열 값을 가짐
    - 측정이 어려운 작업의 경우 다른 값들의 합을 이용하여 어림짐작하기 위함
    - 과거 작업의 기간을 기준으로 새 작업의 기간을 예측
- 플래닝 포커를 수행하는 이유
  - 상대적 측정으로 보다 정확한 작업시간 예측
  - 서로의 지식과 경험, 관점 공유
  - 1개의 스프린트에 진행할 업무량을 결정

- 주의사항
  - 달력의 1일 = 스토리포인트1이 아님!
    - 각 팀, 조직마다 다르며, 내부적 합의가 필요
  - 스토리포인트 1의 의미는?
  - 플래닝 포커는 데드라인인가?
    - 추정일 뿐, 정답이 아님



### 타임박스

- 시간을 끊어 그 단위 안에 어떤 일을 한다는 개념
- 스프린트/플래닝 포커는 타임박스의 개념이 녹아있는 것

- 활용법
  - 오늘 할 일 다 적기
  - 각의 할 일 완료하는데 걸릴 시간 추정
  - 작업 시간 기준으로 하루 스케줄 설정
  - 스케줄대로 일하기

- 활용 시작하기의 각 단계
  - 업무를 최대한 작게 쪼개기
  - 무작정 or 플래닝 포커를 이용하여 작업시간 추정
  - 실제로 얼마나 걸렸는지 작업시간 측정

- 사례
  - 뽀모도로 테크닉
    - 일정 근무 -> 휴식 주기
    - 회사에서 하기에는 어려움
  - KMN
    - 40분 근무 + 20분 주기로 변형: 1KMN
    - 하루에 8KMN을 수행
    - 20분 휴식이 길어 보이지만 실제 업무에서 집중하는 시간이 적음을 감안
    - 휴식 주기가 되면 무조건 바로 쉬러 업무공간을 벗어나는 것이 중요
    - 업무 시간 40분 동안은 집중 근무, 다른 잡 일 전혀 하지 않음
- 타임박스의 특장점
  - 멀티테스킹을 피함
  - 업무 전환으로 인한 낭비가 없음
